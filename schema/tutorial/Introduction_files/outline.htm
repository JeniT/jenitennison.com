XML Schema Fundamentals
All operations are done on infoset of physical document
"elements" rather than "tags"
Schema document is interpreted as a conceptual schema
E.g. xs:element element is interpreted as an element declaration
Multiple schema documents become a single conceptual schema
XML Schema Construction
XML Representations of Schemas
Multiple possible XML representations of the same conceptual schema
Might split up the schema into different physical documents
Might use different element or attribute groups
Constraints on XML Schema documents break down into:
Constraints on the underlying conceptual schema
Constraints on the XML representation of the schema
It's possible to write an XML Schema document that's an OK XML representation, but not an OK conceptual schema
Validation
Physical instance document parsed into XML infoset
Validator augments infoset to create Post-Schema Validation Infoset (PSVI)
Indicates validity/invalidity of nodes
Adds default/fixed attributes & text
Marks "nil" elements
Annotates nodes with data types
Includes schema itself

Schema Components
Target Namespace
Schema components (can) have names
Types can be anonymous, elements and attributes can't
All names are QNames
Have a local part and a namespace URI
All the components in a schema document have the same namespace URI
Known as the target namespace
To validate a document that uses several namespaces, you must have several schema documents
Summary
Different schema languages are useful for different tasks and markup
Schemas have an XML representation, but are defined at a conceptual level
Multiple schema documents combine to create a schema
Validating with a schema creates a PSVI
Schemas contain
Element and attribute declarations
Complex and simple type definitions
Schema documents each have their own target namespace
